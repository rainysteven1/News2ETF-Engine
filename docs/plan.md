# News2ETF 投资策略规划

## 核心思路

**时政新闻 → 行业分类 → 跟踪指数 → ETF 买卖 → 高回报**

---

## 阶段一：新闻数据 → 行业归类

**目标**：将 200 万条时政新闻数据分类到细分行业中。

1. **新闻预处理**
   - 清洗去重、去除无效/广告类内容
   - 提取标题 + 正文关键信息（摘要或关键词）
   - 标注新闻发布时间，用于后续时序对齐

2. **行业分类——分层分类（Hierarchical Classification）**

   采用 **Top-Down 自上而下** 的分层分类架构，而非直接对 28 个行业做平面分类。

   **为什么分层分类更优：**
   - **解决语义混淆**：如"芯片"在平面分类中会在"计算机/电子/通信"间摇摆，分层后先锁定"TMT/科技"大类，过滤 80% 干扰
   - **缓解数据不平衡**：热门行业（AI、新能源）新闻远多于冷门行业（环保、煤炭），分层后各子类仅在大类内部竞争
   - **计算效率更高**：推理时 Level-1 判定为非金融，则无需计算银行/保险等子类

   **层级结构（Level-1 → Level-2）：**

   | Level-1 大类  | Level-2 子类                                                                     |
   | ------------- | -------------------------------------------------------------------------------- |
   | 科技信息      | 半导体/芯片、人工智能、软件/信创、云计算/大数据、通信/5G、物联网/车联网          |
   | 高端制造      | 机器人/智能制造、新能源/光伏、新能源车/锂电、航空航天/卫星、军工/国防            |
   | 医药健康      | 医疗器械/医疗服务、生物医药/创新药                                               |
   | 消费文娱      | 消费电子/家电、食品饮料/消费、传媒/游戏/文娱、农业/食品/畜牧、旅游/养老/社会服务 |
   | 金融地产      | 金融/银行/证券、地产/建筑/基建                                                   |
   | 资源材料      | 能源/油气/资源、有色金属/稀土、化工/新材料                                       |
   | 基础设施/公共 | 交通运输/物流、环保/绿色低碳、ESG/可持续                                         |
   | 主题策略      | 央企/国企/国资改革、区域经济                                                     |

   **模型架构：单模型 + 多分类头（Multi-Head）**
   - 共享一个 FinBERT backbone
   - Level-1 分类头：8 个大类
   - Level-2 分类头：每个大类内 2~6 个子类（共 28 个子类）
   - 联合训练（多任务学习）：`Loss = α × L1_loss + β × L2_loss`

3. **训练数据策略——LLM 标注 + 小模型蒸馏**

   200 万条数据用于训练既无必要也不现实。采用 **关键词预分类 + LLM 兜底 → 小模型蒸馏** 的流水线：

   | 阶段   | 数据量            | 方式                                                                                       |
   | ------ | ----------------- | ------------------------------------------------------------------------------------------ |
   | 第一轮 | 5000~10000 条     | 关键词预分类（高置信度直接标注）+ GLM-4-Flash 兜底标注 Level-1 大类 → 人工抽检 500~1000 条 |
   | 第二轮 | ~50000 条         | 同上流程标注 Level-2 子类 + 情绪，置信度过滤 → 训练 FinBERT 小模型                         |
   | 第三轮 | ~160 万条全量推理 | FinBERT 小模型推理 → 得到 `(新闻, 大类, 子类, 情绪)` 四元组                                |

   **关键词预分类 + LLM 兜底（混合标注方案）：**
   - 先用行业关键词词典对新闻标题做匹配，匹配命中的直接分配标签（覆盖 ~60~70%）
   - 未命中 / 多义词歧义的剩余 30~40% 送 LLM 批量标注
   - 每次 API 调用批量发送 20 条标题（摊薄 system prompt 开销）
   - **好处**：速度提升 2~3 倍、API 调用量减少 60%、关键词规则可解释可调试

   **LLM 标注成本估算（GLM-4-Flash）：**
   - 50000 条标题，批量 20 条/次 = 2500 次 API 调用
   - 关键词预分类后仅需 ~800~1000 次 API 调用
   - 总 token 消耗 ~100 万，成本 **<1 元**

   **质量保障机制：**
   - LLM 标注时同时输出置信度分数，训练时只用高置信度（>0.8）样本
   - 人工抽检 500~1000 条，计算各类别准确率，发现系统性偏差则调整 prompt 重跑
   - 分层抽样：确保冷门行业（环保/绿色低碳、区域经济等）也有足够标注样本

   **为什么还需要训练小模型：**
   - LLM 跑 200 万条推理成本高、速度慢
   - FinBERT 小模型推理成本低几个数量级，可本地 GPU 批量跑
   - 小模型部署简单，适合后续每日增量新闻的实时分类

4. **输出**
   - 每条新闻带有：`新闻ID`、`发布时间`、`大类标签`、`子类标签`、`情绪/利好利空判断`

---

## 阶段二：行业 → 跟踪指数映射

**目标**：根据行业分类，找到对应的 ETF 跟踪指数。

1. **构建行业-指数映射表**
   - 利用已有的 `industry_dict.json`（跟踪指数 → 行业分类的映射）
   - 反向建立：行业 → 跟踪指数列表的查找关系

2. **指数筛选**
   - 同一行业下可能有多个跟踪指数，需根据以下维度筛选：
     - 指数的流动性（对应 ETF 的日均成交额）
     - 指数的市场覆盖度
     - 跟踪该指数的 ETF 数量与规模

3. **输出**
   - 行业 → 推荐跟踪指数（1~3 个）的映射表

## 阶段三：ML/DL 信号层 + Agent 决策层（混合架构回测）

**目标**：ML/DL 负责数值特征提取和时序信号计算，Agent 负责多因素综合研判与仓位决策，在历史数据（2021~至今）上逐日回测。

**核心原则**：LLM 不直接看原始数字——它看不懂时序趋势；ML/DL 把数值处理成结构化信号后交给 Agent，Agent 做语义层面的多因素推理。

---

### 3.1 ML/DL 信号层

ML/DL 的职责是把原始情绪数据转化为**结构化信号向量**，每个行业每天输出一组信号：

| 模型                       | 输入                | 输出信号            | 说明                    |
| -------------------------- | ------------------- | ------------------- | ----------------------- |
| LSTM / Temporal CNN        | 近 N 日行业情绪时序 | 动量分数 [-1, 1]    | 捕捉情绪上升/下降趋势   |
| Isolation Forest / Z-Score | 新闻量时序          | 热度异常分数 [0, 1] | 检测新闻突然放量        |
| LightGBM / XGBoost         | 上述特征拼接        | 综合看多/看空分数   | 多特征融合打分          |
| GNN（可选）                | 行业关联图 + 情绪   | 跨行业传导强度      | 如油价涨 → 新能源车传导 |

**每个行业每日输出的结构化信号向量：**

```json
{
  "industry": "半导体/芯片",
  "date": "2023-06-15",
  "momentum_score": 0.82, // 情绪动量，强看多
  "heat_anomaly": 0.65, // 新闻热度异常，有放量
  "cross_industry_signal": 0.3, // 跨行业传导（可选）
  "composite_score": 0.76, // LightGBM 综合评分
  "trend_direction": "上升",
  "holding_days": 3, // 当前持有天数（来自回测引擎）
  "unrealized_pnl": 0.05 // 当前浮盈
}
```

---

### 3.2 Agent 决策层（LangGraph 工作流）

Agent 接收 ML/DL 生成的结构化信号，进行多因素综合研判：

```
┌──────────────────┐   ┌──────────────────┐   ┌──────────────────┐   ┌──────────┐
│  ML 信号读取节点   │──→│  行业研判节点     │──→│  仓位决策节点     │──→│ 执行节点  │
│                  │   │                  │   │                  │   │          │
│ 从 DuckDB 查询    │   │ 解读信号含义      │   │ 结合持仓状态      │   │ 模拟交易  │
│ 当日各行业信号向量 │   │ 跨行业关联推理    │   │ 输出买卖指令      │   │ 记录日志  │
└──────────────────┘   └──────────────────┘   └──────────────────┘   └──────────┘
```

**Agent 做的事（语义推理层）：**

| 因素            | 来源              | Agent 推理示例                                         |
| --------------- | ----------------- | ------------------------------------------------------ |
| ML 动量信号解读 | LSTM 输出         | "半导体动量 0.82，属于强看多信号"                      |
| 跨行业关联      | GNN 或 Agent 推理 | "油价异常上涨，同时影响能源和新能源车，但方向相反"     |
| 持仓状态判断    | 回测引擎          | "半导体已持有 3 天，浮盈 5%，是加仓还是止盈？"         |
| 多行业竞争      | Agent 推理        | "今日科技和消费都有信号，资金有限，优先配置动量更强的" |
| 宏观风险识别    | Agent 推理        | "多个行业同时利空，市场可能整体回调，降低全仓仓位"     |
| 止盈止损        | 策略参数          | "浮亏超过阈值，强制止损"                               |

**输出格式（每日决策）：**

```json
[
  {
    "action": "buy",
    "industry": "半导体/芯片",
    "etf": "512480",
    "weight": 0.25,
    "reason": "..."
  },
  {
    "action": "hold",
    "industry": "新能源/光伏",
    "etf": "516160",
    "weight": 0.2,
    "reason": "..."
  },
  {
    "action": "sell",
    "industry": "地产/建筑/基建",
    "etf": "159967",
    "weight": 0.0,
    "reason": "..."
  }
]
```

### 3.3 回测引擎设计

- 按交易日逐日遍历历史数据（~1200 个交易日）
- 每日先跑 ML/DL 信号计算 → 再调用 Agent 决策（GLM-4-Flash，temperature=0 保证可复现）
- 模拟资金账户：初始资金、手续费、滑点
- 每日净值计算 → 生成净值曲线
- 所有决策日志（信号 + Agent 推理过程）写入 DuckDB

**成本估算：**

- ML/DL 推理：本地 CPU/GPU 跑，零成本
- Agent 决策：~1200 次 API 调用，GLM-4-Flash **约 2~5 元总计**

## 阶段四：ETF 选择与回测评估

**目标**：将决策信号落地为具体 ETF，评估回测绩效。

1. **ETF 筛选**（同一指数可能有多只 ETF）
   - 优先选择规模大、流动性好、跟踪误差小的 ETF
   - 费率比较（管理费 + 托管费）

2. **回测评估指标**
   - 核心指标：年化收益率、最大回撤、夏普比率、胜率
   - 对比基准：沪深300 / 中证500
   - 分行业归因：哪些行业贡献了主要收益/亏损

3. **决策日志分析**
   - ML 信号 + Agent 推理过程全程可追溯
   - 分析哪些 ML 信号对收益贡献最大（消融实验）
   - 找出 Agent 判断失误的典型模式 → 优化 prompt

---

## 阶段五：策略迭代与优化

1. **ML 模型调优**
   - 调整 LSTM 的时序窗口长度（3日/5日/7日/14日）
   - LightGBM 特征重要性分析，去掉噪声特征
   - 对比加入 GNN 跨行业传导信号前后的收益变化

2. **Agent Prompt 迭代**
   - 分析回测中的亏损交易，找出 Agent 的系统性偏差
   - 调整决策 prompt（如加强止损逻辑、降低追涨倾向）
   - 重新回测，对比策略改进效果

3. **消融实验**
   - 去掉 ML 层，纯 Agent 直接看原始数字 → 对比收益
   - 去掉 Agent 层，纯 ML 规则信号 → 对比收益
   - 验证混合架构的增益来自哪一层
   - 逐一去掉每个 ML 信号，量化各信号贡献
   - 确定最核心的 alpha 来源

## 存储架构

采用 **DuckDB + SQLite3** 双数据库方案，各司其职：

### DuckDB —— 列式存储（业务数据）

适用于大规模、分析密集型的结构化数据，利用列式存储的高压缩率和向量化查询优势。

| 数据表               | 内容           | 说明                                         |
| -------------------- | -------------- | -------------------------------------------- |
| `news_raw`           | 原始新闻       | 新闻ID、标题、正文、发布时间、来源           |
| `news_classified`    | 分类结果       | 新闻ID、大类标签、子类标签、情绪分数、置信度 |
| `industry_sentiment` | 行业情绪聚合   | 行业、日期、情绪均值、新闻热度、情绪趋势     |
| `ml_signals`         | ML/DL 信号     | 行业、日期、动量分数、热度异常、综合评分     |
| `trade_signals`      | 交易信号       | 日期、行业、动作、仓位、来源ETF              |
| `decision_logs`      | Agent 推理过程 | 日期、输入信号、推理文本、输出决策           |
| `backtest_results`   | 回测结果       | 策略ID、日期、持仓、净值、收益率             |

**选择 DuckDB 的理由：**

- 列式存储，200 万行新闻的聚合查询极快
- 嵌入式数据库，无需部署服务，单文件即可
- 原生支持 Parquet 读写，与现有数据格式无缝衔接
- 支持 SQL，便于数据分析和特征工程

### SQLite3 —— 行式存储（实验管理）

适用于小规模、写入频繁的元数据和实验配置。

| 数据表              | 内容     | 说明                                      |
| ------------------- | -------- | ----------------------------------------- |
| `experiments`       | 实验记录 | 实验ID、名称、创建时间、状态、描述        |
| `hyperparameters`   | 超参数   | 实验ID、参数名、参数值、参数类型          |
| `training_metrics`  | 训练指标 | 实验ID、epoch、loss、accuracy、f1_score   |
| `model_checkpoints` | 模型快照 | 实验ID、epoch、checkpoint路径、验证集指标 |
| `label_mapping`     | 标签映射 | 标签ID、大类名、子类名、申万代码          |

**选择 SQLite3 的理由：**

- 实验参数是行级读写，行式存储更合适
- 轻量级，Python 标准库原生支持
- 单文件便于版本管理和实验复现
- 与 W&B 等实验追踪工具互补

### 文件结构

```
data/
├── news2etf.duckdb          # DuckDB 主数据库（新闻、行情、信号）
├── experiments.sqlite3       # SQLite3 实验数据库（参数、指标）
├── converted/                # 原始 Parquet 文件（DuckDB 可直接查询）
└── industry_dict.json        # 行业-指数映射
```

## 关键成功因素

| 因素                | 说明                               |
| ------------------- | ---------------------------------- |
| 新闻分类准确率      | 分错行业 = 买错ETF，直接影响收益   |
| 情绪判断质量        | 利好/利空的判断偏差会导致反向操作  |
| 信号时效性          | 新闻发布后越快生成信号，alpha 越大 |
| 行业-指数映射完整性 | 映射缺失会导致错过投资机会         |
| 仓位与风控          | 再好的信号，没有风控也会亏损       |
